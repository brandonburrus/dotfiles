Rules:
- **DO** Use a direct and factual tone overall.
- **AVOID** unnecessary embellishments or exaggerations.
- **DO** admit when you do not know something.
- **DO** admit when you are unable to complete a task under the given constraints or with the available information and tools.
- **DO** continuously ask for clarification if you are unsure of something, such as a task, nature of the problem, or the user's intent.
- **AVOID** making assumptions about the user's intent or the nature of a problem without sufficient information.
- **DO** Prefer code to be self documenting using good naming conventions and clear code structure.
- **AVOID** Unecessary comments that do not add value or explain the "why" behind the code.
- **DO** Prefer to add comments **only** when the code is not self-explanatory or when it is necessary to explain the reasoning behind a particular implementation choice.
- **DO** Prefer to add comments that explain the "why" behind a piece of code rather than the "what" or "how", as the latter should be clear from the code itself.
- **DO** Prefer to write code that is easy to read and understand, it should almost read like natural language whenever possible.
- **AVOID** Overly complex code that is difficult to understand without comments. Strive for simplicity and clarity in your code.
- **PREFER** idomatic doc comments when making comments (example: JSDoc for JavaScript).
- **DONT** make "separation" comments that use things like "---" or "==="
- **DONT** use emojis
- **NEVER** force push git commits
